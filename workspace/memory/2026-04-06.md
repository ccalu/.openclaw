# 2026-04-06

## S4 — refactor visual consolidado

Hoje o S4 foi finalmente compreendido e consolidado como um sector com shape nova e substancialmente melhor do que a versão anterior.

### Mudança central do sector
O S4 deixou de ter como miolo principal o pipeline textual fragmentado baseado em:
- `research_worker`
- `candidate_evaluator`
- `asset_materializer`

No lugar disso, emergiu um core visual unificado, com duas mudanças estruturais principais:

1. **Target consolidation no boundary S3 -> S4**
   - `target_builder.py` deixou de fazer conversão quase 1:1 das entidades do S3
   - passou a consolidar targets com dedup/contextualização/searchability
   - isto reduziu duplicatas, ambiguidades e targets abstractos fracos

2. **Asset pipeline visual unificado**
   - `s4_asset_pipeline.py` passou a coordenar o miolo do sector
   - substeps actuais:
     - context extraction via GPT-5.4-nano
     - query generation via GPT-5.4-nano
     - image collection via Serper.dev + download paralelo + pHash dedup
     - visual evaluation via GPT-5.4-nano vision

### Actor map real do S4
O S4 ficou menos actor-heavy do que antes.

- `sm_s4_asset_research` continua a existir, mas delega para `supervisor_shell.py`
- `supervisor_shell.py` é agora o orquestrador determinístico real, com checkpoint-resume
- actors OpenClaw activos no miolo final: apenas
  - `op_s4_coverage_analyst`
  - `op_s4_pack_compiler`
- actors deprecated / não canónicos:
  - `op_s4_target_builder`
  - `op_s4_web_investigator`
  - `op_s4_target_research_worker`
  - `op_s4_candidate_evaluator`

### Verdade artefactual
O artefacto canónico do miolo passou a ser:
- `asset_materialization_report.json`

`candidate_set.json` e `evaluated_candidate_set.json` deixaram de ser a verdade central do core visual. Isso inicialmente criou mismatch em coverage/pack (marcando `unresolved` apesar de os assets existirem), mas o problema foi entendido como integração downstream e não como falha do novo core.

### Evidência runtime
O S4 foi provado em múltiplas camadas:
- supervisor E2E completou com sucesso
- resume funcionou
- coverage + pack compilaram
- fluxo S3 -> S4 com target consolidation via helper-direct foi provado E2E
- fluxo W3 -> S3 -> S4 -> B2 completed também foi confirmado

### Leitura estratégica
O S4 agora está num estado muito mais interessante:
- melhor qualidade visual real
- menos dependência em sinais textuais indirectos
- menos fragilidade no miolo
- melhor controlo por substrate Python determinístico

Mas isso também trouxe novos cuidados:
- parte importante do sector migrou de actor para helper-direct
- há uso de API keys hardcoded em Python (OpenAI/Serper)
- isto funciona tacticamente, mas NÃO deve ser tratado como padrão aprovado para sectores futuros

### Extensão importante de 2026-04-07
No dia seguinte, o S4 deu mais um salto estrutural relevante: a layer de `reference readiness` foi realmente implementada e provada em E2E.

O evaluator passou a gerar, para cada asset aprovado:
- um sidecar `.reference_ready.json` ao lado do ficheiro
- e depois um compilado sector-level `s4_reference_ready_asset_pool.json` com grouped views para consumo do S5

Isto resolveu um problema arquitectural importante: assets aprovados por target não bastavam; o S5 precisava de assets mais inteligíveis e seleccionáveis como referências.

### Resultado desta nova layer
- 61 sidecars gerados
- 1 compiled pool gerado
- pipeline E2E continuou rápido e barato (~7m34s, ~$0.14, 93 chamadas GPT-5.4-nano)
- o conteúdo ficou forte para V1:
  - `depicts`
  - `depiction_type`
  - `reference_value`
  - `preserve_if_used`
  - `scene_relevance`
  - `reasoning_summary`

### Leitura arquitectural actualizada
O S4 agora não é só discovery + asset approval.
Também passou a funcionar como provider de uma surface de referência melhor para o S5.

A arquitectura correcta desta frente ficou:
- sidecar por asset = verdade granular local
- compiled pool = selection surface downstream

O refinement pendente principal desta layer não é reabrir arquitectura, mas apenas calibrar melhor o prompt para evitar over-tagging em `reference_value`.

### Próximo trabalho correcto
Antes de abrir o S5 com força, ficou claro que o correcto é absorver o S4 completamente:
- congelar arquitectura canónica
- actualizar docs centrais + docs locais dos agents
- arquivar histórico e marcar legado
- capturar postmortem / troubleshooting / reusable patterns
- actualizar memória e checklists de criação de sectores

### Nova frente aberta em 2026-04-07 — S5 ganha shape real
Depois de consolidar o S4, a discussão avançou fortemente para o S5.

O ganho central foi que o S5 deixou de ser tratado como “visual direction” vaga e passou a ser compreendido como sector de **scene kit design**:
- S4 = grounding factual / referências reais
- S5 = desenho do `scene_kit_spec`
- S6 = materialização do kit
- S7 = resolução editorial da cena

### Peças que ficaram fechadas no S5
1. **Input Assembly V1**
   - baseado no upstream real do Quitandinha
   - fontes: camada narrativa, S3 grounding, S4 reference layer, global direction frame
   - `scene_summary` e `narrative_function` serão derivados na assembly layer
   - `relevant_targets[]` ficam determinísticos em V1
   - `reference_ready_assets[]` vêm deterministicamente do caminho `scene -> linked_target_ids -> reference_ready_asset_pool`

2. **Grammar V1 das famílias**
   - linguagem central: `family_type + family_intent`
   - `family_type` sozinho revelou-se vago demais para o S6
   - camada editorial leve: `editorial_notes` em texto livre

3. **Shape do `scene_kit_spec` V1**
   - fechada em forma compacta, mas forte
   - 6 secções:
     - `scene_core`
     - `scene_direction`
     - `applied_global_constraints`
     - `kit_strategy`
     - `asset_families[]`
     - `delivery_expectations`

### Lições fortes desta fase
- O S5 não deve escolher a composição final da cena; deve preparar o espaço de assets e de possibilidade editorial para o S7.
- O upstream real mostrou dois modos operacionais do S5:
  - cena com suporte de referência do S4
  - cena com pouca/nenhuma referência, onde o kit terá de depender mais de texto + S3 grounding + global frame
- O `video_direction_frame` foi confirmado como peça nova real do S5, e não algo já resolvido upstream.

### Próxima fase correcta
Com input e output do S5 já razoavelmente fechados, o próximo passo passa a ser o **runtime / operational flow do S5**:
- que passos reais o sector executa
- o que é determinístico vs LLM
- como o frame global nasce
- como o input assembly corre
- como o `scene_kit_spec` é produzido e compilado

### Nota pessoal / lição importante
O S4 mostrou com força que materializar actors demais pode piorar a arquitectura real. Quando o miolo precisa de controlo forte de paths, artefactos, checkpoints e validação, helper-direct + substrate determinístico pode ser mais correcto do que insistir num grafo excessivamente agentizado.
