## Research importante sobre OpenClaw runtime / invocation

Houve pesquisa explícita para descobrir como executar agents em runtime no OpenClaw.

### Descobertas principais
#### `openclaw agents`
Serve para **gestão/configuração de agents**:
- add
- list
- delete
- bind
- unbind
- set-identity

Não serve para executar turns.

#### `openclaw agent`
É a superfície real de execução de um turn de agent.

Confirmado via help/docs:
- `--agent`
- `--message`
- `--json`
- `--local`
- `--session-id`
- `--thinking`
- `--timeout`
- `--deliver`

Isto muda o desenho da ponte do S1:
- a integração certa parece ser `openclaw agent`, não `openclaw agents`

#### Modos de runtime confirmados
- gateway-backed
- embedded local (`--local`)

#### Outras superfícies encontradas
- `sessions_spawn` para subagentes/delegação dentro de turns
- `openclaw acp` como bridge mais programática/ACP
- gateway/websocket como caminho mais baixo nível

### Implicação para o S1
A ponte mais promissora para invocar operators do runtime canónico passa a ser:
- `openclaw agent --agent op_s1_scene_director --message ...`
- `openclaw agent --agent op_s1_tts_polish --message ...`

Ainda falta fechar o formato exacto da mensagem file-based, mas o tronco certo da execução já ficou muito mais claro.

## Late-day OpenClaw runtime findings for S1

### `openclaw agent` vs agent registry reality
Depois da pesquisa e de testes controlados:
- `openclaw agent` foi confirmado como a superfície oficial de execução de um turn de agent.
- A hipótese anterior `openclaw agents run ...` estava errada.

Mas os testes também mostraram uma nuance crítica:
- tentar `openclaw agent --agent op_s1_scene_director ...` falhou com `Unknown agent id`.
- `openclaw agents list` mostrou apenas `main (default)`.
- ao mesmo tempo, as pastas dos workspaces do S1 existem em `C:\Users\User-OEM\.openclaw\agents\`:
  - `sm_s1_pre_production`
  - `op_s1_scene_director`
  - `op_s1_tts_polish`
  - `op_s1_tts_polish_semantic_reviewer`

### Durable conclusion
Isto confirmou que:
- **filesystem workspaces != registered OpenClaw agent ids**
- os workspaces do S1 existem no disco, mas **ainda não estão registados/configurados no registry runtime do OpenClaw CLI**
- portanto o próximo missing link para o S1 canónico não é mais arquitetura nem superfície de execução, e sim **registry/configuration dos agents do S1**

### Immediate implication for future work
O próximo passo correcto depois desta sessão é investigar:
- `openclaw agents add --help`
- onde o OpenClaw guarda a registry/lista real de agents
- como transformar um workspace existente em agent id invocável pelo `openclaw agent --agent ...`

### Additional implementation note
Foi ajustado `runtime/s1/supervisor_runtime.py` para:
- usar `openclaw agent` no hook `_invoke_scene_director_batch(...)`
- criar `_build_scene_director_agent_message(...)` com convenção file-based (input payload path -> output artifact path)

Mas esse hook ainda depende da resolução do problema acima (agent ids do S1 não registrados).
