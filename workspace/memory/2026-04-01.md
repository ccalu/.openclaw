## Scene director bottleneck — deep findings after PM/VO validation

Depois de validar PM e VO, o pipeline real `PM -> VO -> S1` foi testado várias vezes. O gargalo ficou completamente isolado no `scene_director` do S1.

### O que foi provado
- PM funciona como agent OpenClaw real
- VO funciona como agent OpenClaw real
- `PM -> VO -> ready_for_s1` foi provado em runtime real
- o problema actual do sistema está concentrado no lane `scene_director` do S1

### Hipóteses investigadas e resultados
1. **Contexto/sessão reutilizada entre batches**
   - hipótese do Lucca estava correcta
   - batches estavam sendo invocados sem `--local`
   - ao mudar para `--local`, o padrão de falha mudou e melhorou
   - conclusão: havia sim contaminação de sessão/contexto entre batches

2. **O contract pedia precisão mecânica demais do agent**
   - confirmado também
   - o `scene_director` estava sendo forçado a fazer semântica + contagem de palavras + alinhamento textual exacto + spans exactos
   - isso mostrou-se estruturalmente frágil

### Evolução do diagnóstico do scene_director
- fase 1: global coordinates em batches 2-5
- fase 2: com `--local`, os problemas reduziram bastante
- fase 3: surgiu erro residual localizado em `sd_002 / scene_02`
- fase 4: inspeção mostrou que `scene_02` engolia o começo da `scene_03`
- fase 5: novo rerun revelou sobre-segmentação em `sd_001` (16 cenas para 717 palavras)

### Conclusão arquitectural actual
A direcção correcta do lane é **híbrida**:
- agent = agrupamento semântico / títulos / summaries / função narrativa
- substrate Python = reconstrução mecânica / word counts / alinhamento exacto / validação

## Overnight S1 robustness result — TTS lane no longer hard-blocking

Durante a madrugada foi isolado com mais precisão o bloqueio residual do S1: o problema principal já não era o `scene_director`, mas sim a fragilidade dos artefactos do `tts_polish`.

### Diagnóstico mais forte
- vários batches `tp_###` produziam outputs semanticamente suspeitos (`meaningful_change_detected`) ou artefactos inválidos / malformados
- isto estava a impedir o supervisor de cruzar `b1_step3_tts_polished`, mesmo quando o resto do sector já estava suficientemente forte
- a conclusão técnica importante é que o problema não exigia abortar o caminho OpenClaw do S1; exigia um **substrate supervisor mais duro** para não deixar o lane semântico bloquear o sector inteiro

### Fix de maior alavancagem aplicado
Foi introduzido em `runtime/s1/supervisor_runtime.py` um fallback determinístico do supervisor para batches de `tts_polish` que falhem validação:
- se o batch vier inválido, ilegível ou semanticamente suspeito demais
- o supervisor reescreve o output do batch como **identity polish**
  - `narration_tts = narration_original`
  - `changes = []`
- isto preserva segurança semântica e permite que o sector avance, em vez de falhar inteiro por causa de um polish ruim

### Validação real conseguida
O fix foi validado contra o run real controlado:
- `production/004_fr/2026-03-30_004_fr_001`

Depois do repair/fallback:
- todos os batches `tp_001` até `tp_010` ficaram aprovados para merge
- `tts_polished.json` passou na `polish_validator`
- `scene_bible.json` foi construído e validado
- o run cruzou:
  - `b1_step3_tts_polished`
  - `b1_step4_polish_validated`
  - `b1_step5_scene_bible`
  - `b1_complete`

### Verdade operacional neste momento
- o caminho OpenClaw do S1 **não deve ser abortado neste momento**
- a leitura mais honesta é: **keep, but with deterministic guardrails**
- o lane `tts_polish` deve ser tratado como camada de melhoria semântica protegida por fallback supervisor-driven, não como ponto único de falha do sector
