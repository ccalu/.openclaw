## Mudança central: abandono do foco em S1/S2, viragem para Bloco 2

### O que percebemos

Depois de mais de 3 dias a tentar fazer o S1 funcionar com OpenClaw, o Lucca trouxe uma conversa que teve com o ChatGPT esta manhã (`luccaechatreformopenclaw.txt`), onde ficou muito claro o que estava a correr mal:

**estávamos a agentificar partes do sistema que já funcionavam.**

O S1 e o S2 — que correspondem ao Bloco 1 do CFV3 — já são um pipeline híbrido sólido:
- agente faz o que é semântico/ambíguo
- Python corrige, valida e garante mecanicamente

Gastar semanas a reimplementar isso em agentes OpenClaw é overengineering puro.

### O que deve realmente ser construído com agentes

O verdadeiro problema da Content Factory é o **Bloco 2 — produção visual**.

O CFV3 perdeu acesso às APIs de imagem Google, e as alternativas de API são caras e limitadas.

A solução real é **browser automation (Playwright) em plataformas com subscrição mensal** como Freepik e Higgsfield.

Isso sim é um problema que pede agentes: exploração, julgamento visual, retries qualitativos.

## Nova arquitectura geral

Ficou decidido que o novo sistema tem esta lógica:

- **Bloco 1** → preservar o CFV3 tal como está; expor como mini-orquestrador invocável
- **Bloco 2** → construir com agentes OpenClaw + Playwright (foco principal nosso)
- **Bloco 3** → preservar o CFV3 tal como está; expor como mini-orquestrador invocável
- **Paperclip** → orquestrador global que coordena os blocos

### Mini-orquestradores (workflows paperclip-facing)
O conceito central do novo sistema:
- em vez de 1 orquestrador Python monolítico que controla tudo
- existem vários **mini-orquestradores** invocáveis pelo Paperclip
- cada um controla uma fatia delimitada do pipeline
- o Paperclip decide quando chamar e em que ordem

Nomenclatura provisória discutida:
- `w1_screenplay.py`
- `w2_tts_audio.py`
- `w3_visual.py` (a construir para o Bloco 2)
- `w4_assembly.py`
