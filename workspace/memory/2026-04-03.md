## 1. Decisão central do dia: B2 como unidade paperclip-facing

Ficou decidido que o **Bloco 2** deve ser tratado como **uma unidade paperclip-facing única**.

Isto significa:
- o Paperclip não deve orquestrar S3, S4, S5 e S6 separadamente
- o interior do B2 deve ser orquestrado no OpenClaw
- o Paperclip deve ver o B2 como um bloco único com input, output e estado macro

### Consequência arquitectural

O modelo recomendado passou a ser:

```text
Paperclip
  -> boundary workflow do B2
    -> b2_director
      -> supervisores sectoriais
        -> operators
```

---

## 2. `b2_director` definido como control plane interno do B2

Foi fechado que o `b2_director` é o director interno do Bloco 2 no OpenClaw.

### Papel do `b2_director`
- ler o estado actual do bloco
- interpretar o evento mais recente
- decidir o próximo sector correcto
- activar o supervisor apropriado
- actualizar `b2_state.json`
- escrever checkpoints do bloco
- fechar o bloco em `completed` ou `failed`

### Limites importantes
O `b2_director` **não** deve:
- executar trabalho interno dos sectores
- chamar operators directamente
- monitorizar operators directamente
- depender de conversa longa acumulada

Foi criada documentação central do director e também o workspace local base do agent.

---

## 3. Runtime do B2 tornado tangível

Foi documentado o **B2 runtime control flow** com foco em tornar o sistema realista e operacional.

### Ideia principal
O B2 deve funcionar como uma **state machine orientada a artefactos e reentradas discretas**.

Ou seja:
- estado persistido no disco
- checkpoints por sector
- activações discretas do director
- nada de agent "sempre acordado" dependente de conversa contínua

### Pergunta que isto resolveu
Como é que o director “espera” e “monitoriza” sem uma sessão viva contínua?

Resposta:
- ele não espera mentalmente
- ele é reactivado com base em checkpoints e estado persistido

## 10. Investigação importante: sessões persistentes thread-bound no WebChat/Nerve

O Lucca levantou a necessidade de `mode: "session" + thread:true` para trabalho continuado.

Isto foi investigado em profundidade, incluindo:
- docs locais do OpenClaw
- estado do runtime
- pesquisa paralela com subagentes
- pesquisa do Claude Code

### Conclusão
**WebChat / Control UI / Nerve não suportam hoje `sessions_spawn` persistente com `mode:"session" + thread:true` para subagents normais.**

Isto parece ser limitação real de channel/plugin hooks, não uma config escondida simples.

### Consequência prática
O B2 não deve depender dessa primitive.

A boa notícia é que o desenho do B2 já estava correcto:
- `mode: "run"`
- estado persistido no disco
- reentradas discretas
- checkpoints

Ou seja: o B2 está alinhado com a realidade da plataforma.

## Estado emocional/estratégico implícito

Hoje houve uma transição importante:
- menos discussão abstracta
- mais programa real de execução
- mais alinhamento entre arquitectura e as limitações reais do runtime

A sensação correcta no fim desta fase é:
- ainda há bastante implementação pela frente
- mas o desenho central do B2 está muito mais sólido
- e já existe uma linha clara até ao primeiro teste do B2 com S3-only
- B2/S3 runtime validation reached a decisive point: the correct primitive for `supervisor -> operator` in the current environment is **`exec -> openclaw agent --agent <operator_id> --message ... --json`**; ACP/sessions_spawn should not be treated as the central orchestration substrate for this path.
- `b2_director` is now materially proven in runtime real for: (1) active bootstrap, (2) idempotent/no-op bootstrap, and (3) resume `s3_completed -> b2_completed` in `mode = s3_only`.
- The correct B2 split is now explicit: `b2_director` is the macro decision-maker and writes checkpoints like `s3_requested.json`; `w3_block2.py` is the boundary observer/executor that detects those checkpoints and invokes `sm_s3_visual_planning`; the supervisor then invokes operators.
- All 4 S3 operators were proved in real runs via the canonical CLI primitive. The key lessons were: `environment_location_extractor` and `symbolic_event_extractor` initially wrote legacy output shapes and had to be aligned to canonical `entities[]`-first schemas; `object_artifact_extractor` initially failed because dispatch generation was missing; `human_subject_extractor` was the first fully clean proof.
- The first real external E2E-style Paperclip-side test exposed 3 last-mile production bugs in the supervisor path: (1) regression to ACP / subset execution, (2) failure to mirror `s3_completed.json` or `s3_failed.json` into `{b2_root}/checkpoints/`, and (3) path/encoding drift (`—` vs `-`) causing writes into the wrong directory tree. These are OpenClaw/supervisor-side bugs, not `w3_block2.py` bugs.
- Supervisor hardening was applied in docs/instructions so that it should: (1) use CLI launch instead of ACP, (2) avoid silent subset execution, (3) write macro checkpoints into `{b2_root}/checkpoints/`, and (4) treat bootstrap paths as literal/canonical with no heuristic reconstruction.
- A central postmortem and reusable gate now exist in `content_factory_block2/POSTMORTEM_AND_LEARNINGS_2026-04-03.md` and `content_factory_block2/SECTOR_PREFLIGHT_CHECKLIST.md`. These should be treated as mandatory references before opening any future sector (S4/S5/S6).
- New production-hardening lesson: `openclaw agent --agent ...` reuses sessions by default, which will cause context/token buildup across videos. For production, fresh/isolated run behavior must be enforced instead of relying on default session reuse.
- Recommended current workaround for clean context per run: use unique `--session-id` values per run/stage when invoking agents via CLI. Boundary-side (`w3_block2.py`) hardening was applied conceptually for director/supervisor stages, and the same principle must also hold for `sm_s3_visual_planning -> operators`.
- Current go/no-go judgment for Perseus evolved during the day: after director + supervisor + operator-layer proofs, B2 became ready enough for a **narrow S3-only Paperclip/Perseus integration test**, but only as a checkpoint-driven, artifact-driven run — not as proof that the full block is production-hard yet.
- Important communication/operational lesson: do not say “agent created” when only a folder/workspace exists. Only call an agent materialized when it appears in `openclaw agents list` and executes a real turn.
